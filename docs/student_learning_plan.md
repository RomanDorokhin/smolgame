### План обучения для студента: Чего не хватило для автономного деплоя

Чтобы в следующий раз студент выдавал билд, который деплоится сам, его нужно научить следующим 4 пунктам. Этот план предназначен для базы знаний (KIs) или системного промпта.

#### 1. Внедрение цикла валидации (Validation Loop)
**Проблема:** Студент пишет код и считает задачу выполненной, не проверяя, компилируется ли он.
**Чему научить:**
*   После каждой генерации сцены или сущности **обязательно** запускать `npm run build` или `npx tsc --noEmit`.
*   Если есть ошибки — не игнорировать их, а исправлять по одной, анализируя трейс ошибки.

#### 2. Строгое соблюдение протокола Base-классов
**Проблема:** Студент путал абстрактные шаблоны (`_TemplatePlayer`) и базовые логические классы (`BasePlayer`), а также забывал про модификаторы `public/protected`.
**Чему научить:**
*   **Чтение сигнатур:** Перед написанием `super()` в конструкторе, он должен заглянуть в файл родительского класса и проверить структуру `config`.
*   **Документация шаблона:** Студент должен всегда сверяться с `docs/modules/top_down/template_api.md` (или аналогом), где прописаны правила переопределения методов.

#### 3. Синхронизация ассетов (Asset Integrity Check)
**Проблема:** Студент придумывал названия текстур в коде (например, `enemy_zombie`), в то время как в `asset-pack.json` они назывались иначе.
**Чему научить:**
*   **Single Source of Truth:** Перед использованием любого `textureKey`, студент должен прочитать `assets/asset-pack.json` и использовать только те ключи, которые там прописаны.
*   Если ключа нет — он должен сначала добавить его в конфиг ассетов, а потом в код.

#### 4. Знание специфики движка (Phaser + TS)
**Проблема:** Студент допускал ошибки в базовых конфигах Phaser (например, указывал только `y` для гравитации, когда TS требовал и `x`).
**Чему научить:**
*   Внедрить в него «библиотеку типичных фиксов» для этого конкретного движка.
*   Научить проверять `gameConfig.json` на соответствие интерфейсу `GameConfig` из Phaser.

#### 5. Приоритет механики над визуалом (Mechanics First)
**Проблема:** Огромная работа была проделана над дизайном (прорисованы герои, анти-герои, локация), но **игровой механики и физики нет вообще**. Игрок появляется "в кубике" и ничего не может сделать. Игра даже не стартовала без ручного вмешательства.
**Чему научить:**
*   **Игровой цикл (Game Loop) — основа.** Студент не должен увлекаться визуалом и ассетами, пока не реализована **базовая играбельность**: передвижение, физика, взаимодействия.
*   Акцент должен быть на том, чтобы "игра была игрой". MVP игры — это когда пользователь может совершать осмысленные действия, даже если вместо персонажей просто геометрические фигуры. Визуал натягивается только на работающую механику.

> `Run 'npm run build' -> Parse errors -> Fix files -> Repeat until success -> Run deploy script.`
> Сейчас он пропускает шаги "Parse errors" и "Repeat", надеясь на удачу.
> **Дополнительно:** добавить обязательный шаг дымового тестирования (Smoke Test), чтобы убедиться, что базовое движение и физика работают.
